# 企微消息读取问题分析

## 核心问题

**无法读取到最后发送的消息**

### 现象
- 用户在企微中发送消息后
- 使用 macOS Accessibility API 读取聊天记录
- 读取到的消息列表缺少最后 1-2 条刚发送的消息
- 即使等待 2 秒后读取，仍然缺失

### 示例
**实际聊天记录（屏幕上可见）：**
```
17. [客户] ok
18. [客户] 请问一下...
19. [我方] @客户 我确认下
20. [我方] 好的，稍后回复
```

**程序读取到的：**
```
17. [客户] ok
18. [客户] 请问一下...
（缺少 19、20 两条）
```

## 技术背景

### 当前实现方式
使用 macOS Accessibility API 读取企微聊天窗口：
1. 获取焦点窗口 (AXFocusedWindow)
2. 遍历查找 AXTable 元素
3. 读取 AXTable 中的 AXRow 元素
4. 提取每行的文本内容

### 代码片段
```python
def get_messages(focused):
    # 先滚动到底部
    scroll_to_bottom()
    
    # 查找所有 AXTable
    tables = walk_collect(focused, lambda el: role(el) == 'AXTable', max_depth=10)
    
    # 选择行数最多的 table
    scored = []
    for table, path in tables:
        rows = walk_collect(table, lambda el: role(el) == 'AXRow', max_depth=3)
        score = len(rows)
        scored.append((score, rows))
    
    scored.sort(key=lambda x: x[0], reverse=True)
    all_rows = scored[0][1]
    
    # 只取最后 20 条
    recent_rows = all_rows[-20:]
    
    # 解析消息
    for row in recent_rows:
        tokens = flatten_texts(row, max_depth=6)
        messages.append({'sender': tokens[0], 'content': tokens[1:]})
    
    return messages
```

## 已尝试的解决方案

### 方案1：增加等待时间 ❌
- 检测到新消息后等待 1 秒 → 失败
- 增加到 2 秒 → 仍然失败
- **问题：** 等待时间不是根本原因

### 方案2：滚动到底部 ❌
- 读取前模拟 Cmd+Down 滚动到底部
- 等待 0.3 秒让 UI 更新
- **问题：** 仍然读取不到最后的消息

### 方案3：选择行数最多的 table ❌
- 优先选择包含最多行的 AXTable
- **问题：** 最新消息可能还没加载到 table 中

### 方案4：只取最后 20 条 ❌
- 从所有行中取最后 20 条
- **问题：** 如果总行数就不包含最新消息，取最后 20 条也没用

### 方案5：二次读取 ⏳（当前方案）
- 检测到消息变化后等待 2 秒
- 重新读取一次消息
- **状态：** 待测试

### 方案6：Accessibility Notification 回调 ⏳（实验中）
- 使用事件驱动而不是轮询
- 监听 kAXValueChangedNotification 等通知
- **状态：** 已实现，待测试效果

## 根本原因分析

### 可能的原因
1. **UI 渲染延迟**
   - 企微发送消息后，UI 元素需要时间渲染
   - Accessibility API 读取的是已渲染的元素
   - 最新消息可能还在渲染队列中

2. **AXTable 更新延迟**
   - 聊天记录存储在 AXTable 中
   - 新消息可能还没添加到 table 的 children 列表
   - 即使 UI 上可见，Accessibility 树可能还没更新

3. **虚拟滚动/懒加载**
   - 企微可能使用虚拟滚动优化性能
   - 只有可见区域的消息才会加载到 Accessibility 树
   - 最新消息可能在"即将可见"的缓冲区

4. **多个 AXTable**
   - 企微可能有多个 table（历史消息、当前消息等）
   - 最新消息可能在另一个 table 中
   - 当前逻辑只选择行数最多的一个

## 需要探索的方向

### 方向1：深入理解企微 UI 结构
- 使用 Accessibility Inspector 工具实时观察
- 发送消息前后对比 UI 树的变化
- 找到最新消息实际存储的位置

### 方向2：尝试其他 Accessibility 属性
- 除了 AXTable/AXRow，是否有其他元素类型？
- 是否有 AXScrollArea 或其他容器？
- 是否可以直接访问"最后一条消息"的元素？

### 方向3：组合多种触发机制
- 不依赖读取到"最后一条"
- 只要检测到消息变化就触发判断
- 用 AI 判断当前可见的消息是否足够做决策

### 方向4：使用 OCR 或截图
- 如果 Accessibility API 不可靠
- 考虑截图 + OCR 识别最后几条消息
- 或者使用 Vision Framework 识别文本

### 方向5：Hook 企微进程
- 更底层的方案：hook 企微的消息发送/接收函数
- 直接从内存中读取消息数据
- **风险：** 可能违反企微使用协议

## 当前最佳实践建议

基于现有限制，建议采用"容错策略"：

1. **不强求读取到最后一条**
   - 接受可能有 1-2 条消息的延迟
   - 依靠 AI 判断当前可见的消息

2. **增加判断频率**
   - 缩短轮询间隔（1 秒）
   - 或使用 Notification 回调

3. **多次尝试**
   - 检测到变化后，间隔 1 秒读取 3 次
   - 取读取到消息最多的那次

4. **结合超时机制**
   - 如果最后一条是我方消息
   - 20 分钟无新消息自动关闭
   - 不依赖读取到客户的回复
